## 1. 问题诊断

### 1.1 当前状态

工作流 v2 架构（6 Agent + 3 基础设施）设计质量高，但存在一个根本性局限：

**所有 Agent Prompt、Protocol Schema、模板均硬编码为"玄幻/废材逆袭/林云"单一题材。**

这意味着换一个题材（如都市悬疑、仙侠言情）需要大量重写 Agent Prompt。

### 1.2 硬编码影响范围（待 Phase 0 完整扫描）

| 模块 | 硬编码内容 | 影响 |
|------|-----------|------|
| Advisor Prompt | Q1-Q5 问题偏向玄幻、Genre Profile YAML 字段为玄幻设计 | 无法引导其他题材的创意收敛 |
| Planner Prompt | ChapterMemo 示例为废材逆袭节奏 | 大纲节奏不适合其他题材 |
| Writer Prompt | 爽点模式、战力体系引用 | 写出都市文会带玄幻味 |
| Auditor Prompt | 40 维审计中部分维度题材绑定 | 对悬疑文的悬念密度要求不同于玄幻 |
| Protocols | chapter_memo 示例、audit_report 维度 | 数据结构耦合题材假设 |
| Templates | 黄金开头、情绪曲线以玄幻为例 | 悬疑/言情需要不同的节奏曲线 |

### 1.3 根因

当前缺少 **题材无关的通用引擎 + 可插拔 Genre Profile** 分层设计。题材配置散落在各 Agent Prompt 中，而非集中在一个外置配置文件中，由 Agent 启动时注入。

### 2.2 inkos 的 15 Genre Profile（skills-analysis/inkos.md 已拆解）

inkos 是目前业内最完善的多题材配置参考。每个 Genre Profile 包含：

- 题材底色（世界观类型、力量体系、科技水平）
- 数值系统（战力等级/修为境界/货币体系）
- 爽点节奏（战斗密度、突破频率、打脸间隔）
- 疲劳词表（该题材最容易滥用的词汇）
- 禁止模式（该题材的写作禁忌）
- 平台期望（Royal Road / Amazon Kindle 的类型约定）

**但 inkos 的 Genre Profile 是英文网文向的**，需要转化为中文网文体系（玄幻/都市/仙侠/悬疑/言情/种田/系统流/无限流等）。

## 3. 重构目标

将工作流重构为 **题材无关的通用引擎 + 可插拔 Genre Profile**：

```
用户选择题材
      ↓
Genre Profile 加载（外置 YAML/Markdown 文件）
      ↓
注入所有 Agent（Advisor/Planner/Writer/Auditor/Reviser/Polisher）
      ↓
同一套流水线，不同题材输出不同风格和节奏
```

### 3.1 核心原则

1. **Agent 逻辑与题材数据分离**：Agent Prompt 只描述"怎么做"，Genre Profile 描述"做什么题材"
2. **Genre Profile 是单一事实来源**：所有 Agent 从同一个文件读取题材配置
3. **向后兼容**：现有玄幻废材逆袭数据（第 1-10 章）作为默认 Profile，重构后输出等价
4. **可扩展**：新增题材只需新建一个 Genre Profile 文件，无需修改 Agent

- 不引入新的外部依赖或编程语言——保持纯 Markdown/YAML/JSON Schema
- 不一次性重构所有文件——严格按 Phase 0 → 1 → 2 → 3 顺序

## 9. 与讨论文档1 的关系

讨论文档1 定义了 v2 工作流架构（6 Agent + 3 基础设施），本文档在该架构之上进行**题材通用化改造**，不推翻核心流水线设计，而是：

- 将散落在各 Agent 中的题材配置抽取为外置 Genre Profile
- 使同一套 Agent 流水线可服务于不同题材
- 消除"换题材 = 重写 Agent Prompt"的痛点
