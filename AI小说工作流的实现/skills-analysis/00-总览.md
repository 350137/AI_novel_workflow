## 三、Top 3 推荐组合方案

### 方案一：最强基础设施组合（推荐用于搭建自有工作流）

```
底层：onkos（记忆管理 + 事实追踪 + 6级摘要 + 快照回滚 + 确定性审查）
中层：my-novel-writer（CLI 生成引擎 + 风格锁定 + 字数控制）
上层：novelist（句法红线 + 文字淬炼 + 推镜法则）
参考：inkos（ChapterMemo 7段 + Governed Context + 审/改/润分离 + Hook模型）
```

- **优点**：每个维度都取最佳实现；onkos提供最可靠的底层基础设施
- **缺点**：集成复杂度最高；需自建编排层串联各组件
- **适合**：有开发能力，想搭建自有工作流的团队

### 方案二：最小可行组合（推荐用于快速启动）

```
策划端：fiction-crafter（8维创意 + 大纲 + 9维审查）
写作端：my-novel-writer（章节生成 + 风格锁定 + 字数控制 + 伏笔追踪）
```

- **优点**：2个 skill 覆盖创作全流程；集成成本最低；都是中文网文验证过的
- **缺点**：无本地NLP（纯LLM）；审查维度不如组合方案一全面
- **适合**：个人作者，快速开始写小说

### 方案三：番茄平台特化组合

```
策划端：novelist（人物深度 + 句法红线）+ fiction-crafter（大纲 + 审查）
写作端：tomato-novelist（平台规范 + 情绪标签 + 数据反馈环 + 钩子轮换）
数据层：onkos（记忆管理 + 事实追踪，未来可选）
```

- **优点**：tomato-novelist 是唯一有平台数据反馈环的 skill；文风控制最强（novelist 句法红线 + tomato 番茄味润色）
- **缺点**：仅番茄平台；onkos 集成需额外开发
- **适合**：专注番茄小说平台的创作者

## 四、双终端分工建议

基于上述分析，推荐以下双终端分工方案：

### Terminal A：策划与审查端

| 组件 | 来源 | 职责 |
|------|------|------|
| 创意顾问 | fiction-crafter（复用8维生成） | 阶段〇+一：头脑风暴→8维完整设定 |
| 大纲规划师 | 新建（参考 inkos ChapterMemo） | 阶段二：树状大纲+结构化章节指令 |
| 伏笔追踪器 | onkos hook_tracker（基础）+ inkos Hook模型（升级） | 贯穿线：dependsOn/coreHook/promoted/urgency |
| 质量审计官 | onkos 脚本审查（Layer1）+ 新建 LLM审计（Layer3） | 阶段四：40维三层审计 |
| 记忆管理器 | onkos（SQLite+6级摘要+快照+事实引擎） | 贯穿线：数据存储和上下文检索 |

**Terminal A 工作节奏**：触发式——有策划任务时工作，产出存入共享目录。每批章节完成后执行质量审计。

### Terminal B：写作与润色端

| 组件 | 来源 | 职责 |
|------|------|------|
| 章节生成器 | my-novel-writer（生成引擎）+ novelist（句法红线+推镜法则） | 阶段三：基于 ChapterMemo 逐章生成 |
| 修订官 | 新建（参考 inkos LOCAL/STRUCTURAL 路由） | 审计后修补或重写 |
| 润色官 | novelist（文字淬炼）+ tomato-novelist（平台润色） | 文笔层面最终打磨 |
| 平台适配器 | tomato-novelist（番茄）/ 新建（其他平台） | 平台规范+数据反馈 |

**Terminal B 工作节奏**：持续运行——自动生成→审查退回→修订→润色→落定。每10-20章暂停等人类抽查。

## 五、生态空白地带汇总

以下是当前所有 10 个分析对象都未覆盖的需求——需要自行开发：

| # | 空白领域 | 严重程度 | 建议 |
|---|---------|---------|------|
| 1 | **Agent 编排层（Orchestrator）** | 🔴 关键 | 串联所有组件的流水线引擎——这是最高优先级的新建项 |
| 2 | **自动设定矛盾检测** | 🔴 关键 | 写作后自动检测"本章是否违反世界观规则"，onkos continuity 最近似 |
| 3 | **因果链自动验证** | 🔴 关键 | 检测"事件A是否必然导致事件B"的因果合理性 |
| 4 | **中式网文 Genre Profile** | 🟠 重要 | inkos 有15个英文类型预设，但缺中文玄幻/都市/仙侠的完整配置 |
| 5 | **多平台发布策略** | 🟠 重要 | 同一小说差异化适配不同平台（起点/番茄/微信读书） |
| 6 | **创作分析仪表盘** | 🟠 重要 | 进度/质量趋势/伏笔回收率/角色出场频率的可视化面板 |
| 7 | **跨章时间一致性检查** | 🟡 中等 | 自动检测"从第3章到第8章经过了多长时间，角色年龄是否正确" |
| 8 | **角色弧光自动检查** | 🟡 中等 | 自动检测"角色成长弧线是否完整" |
| 9 | **多版本草稿管理** | 🟡 中等 | 同一章保留多个版本文本并比较 |
| 10 | **风格迁移** | 🟡 中等 | 学习作者A的风格→应用到小说B |
| 11 | **增量生成** | 🟢 低 | 骨架→对话→环境→润色的逐层生成模式 |
| 12 | **成本追踪** | 🟢 低 | 追踪每次LLM调用的token消耗和费用 |
| 13 | **A/B测试框架** | 🟢 低 | 不同版本在不同平台测试读者反应 |
| 14 | **读者画像分析** | 🟢 低 | 根据在读读者的偏好数据自动调整创作策略 |
