## 1. 项目起源

### 1.1 初始动机

用户准备搭建一个基于 Claude Code 的 AI 小说工作流，实现多 Agent 相互协作完成长篇小说创作。当时已完成前期准备——安装了一系列技能（skills）作为工作流的组件储备。

### 1.2 初始想法（未优化前的原始描述）

```
1. 搭建AI小说工作流，基于Claude Code，实现多Agent相互协作
2. 安装的一系列技能均是为搭建工作流做准备
3. 目前处于想法初级阶段，需要进行头脑风暴
4. 任务分解：
   (1) 前期想法：题材、创意、故事大体框架
   (2) 确定题材和创意后，搭建世界观、战力体系
   (3) 设定主角配角，单独文档记录各角色剧情和设计，创建人物图谱
   (4) 开始构建故事，规划每个章节的具体内容
```

### 1.3 可用技能清单

| 技能 | 类型 | 在工作流中的潜在角色 |
|------|------|---------------------|
| fiction-crafter | 创意生成 | 8维创意提示词（题材/世界观/主角/冲突/爽点/节奏/配角/开篇钩子） |
| my-novel-writer | 章节生成 | OpenAI兼容API调用 + 本地JSON记忆管理 |
| story-cog | 故事认知 | CellCog API封装（远程智能） |
| onkos | 本地NLP | SQLite FTS5 + jieba分词 + ONNX语义模型（73条命令） |
| prompt-optimizer | 提示词优化 | 6阶段分析流水线（项目检测→意图检测→范围评估→ECC组件匹配→缺失上下文检测→工作流推荐） |

### 1.4 技能分类分析

经讨论，三个关键技能的本质差异被厘清：

| 技能 | 依赖 | 本质 |
|------|------|------|
| story-cog | CellCog 远程 API | 纯远程智能封装，零本地推理 |
| my-novel-writer | OpenAI 兼容 API | API调用 + 本地JSON记忆（pydantic配置 + prompt工程） |
| onkos | 本地 NLP 引擎 | jieba 词性标注 + ONNX text2vec 语义向量（768维）+ SQLite FTS5 全文检索 |

**关键认识**：本地 NLP（onkos）的价值在于离线实体识别和语义检索，不依赖外部 API，适合敏感内容处理和快速本地索引。远程 API（story-cog、my-novel-writer）的价值在于生成能力和故事理解。

## 2. 实战验证：手动跑通完整流水线

在架构讨论之前，先用 fiction-crafter 实际跑了一遍完整流程，以"废材逆袭流玄幻小说"为题材，完成以下步骤：

### 2.1 创意生成（5轮迭代）

每轮迭代聚焦不同维度：

| 轮次 | 讨论焦点 | 关键决策 |
|------|----------|----------|
| 第1轮 | fiction-crafter 初始8维输出 | 书名空缺、主角名待定、双女主(B) + 群像(B)、开篇钩子选C |
| 第2轮 | 节奏架构修正 | 否决"每1-2章打脸"的快节奏——引入"因果链节奏模型"和"六种节奏模式交替" |
| 第3轮 | 挫折设计、修罗场 | 大爽点四要素（对手之强+差距之悬+临阵领悟/意志力+真实代价），双女主四大矛盾线 |
| 第4轮 | 区域境界天花板 | 东荒最高7境（天象）、中州最高8境（圣王）、帝境（9境）万年无人 |
| 第5轮 | 东荒篇幅扩展 | 东荒从~180章扩展到~240章（4阶段×3-4子阶段×3-5情节单元，全树状结构） |

### 2.2 最终创意产出

- **书名**：暂缺
- **主角**：林云，灵脉被封印的废材，混沌珠宿主
- **双女主**：沈清雪（青云宗宗主之女，清冷医道）+ 姬瑶（妖族公主，傲娇情报网）
- **总篇幅**：~750章（东荒240 + 中州220 + 天道170 + 超脱120）
- **境界体系**：9境（炼体→开元→凝魂→灵海→神府→天象→道台→圣王→帝境）+ 超脱
- **节奏模型**：因果链驱动（打脸→引起注意→引出新人物→矛盾升级→对手太强→自然挫折→反弹）+ 六种模式交替（爽文/拉扯/憋屈蓄力/爆发/缓冲/悬念）
- **产出文件**：`提示词.md`（~1000行完整创意设定）

### 2.3 大纲规划

基于创意提示词，生成全局大纲：

- **第一卷**（东荒，1-240章）：逐章任务分解，每章标注核心任务/节奏模式/伏笔
- **第二至四卷**（中州/天道/超脱，241-750章）：阶段级分解
- **伏笔追踪表**：17条核心伏笔，标注埋设章→发展提示章→回收章
- **节奏分布分析**：六种模式各自占比
- **产出文件**：`大纲规划.md`（~750行完整大纲）

### 2.4 章节生成（第1-10章）

逐章生成正文，每章2200-2500字：
- 第一人称视角
- 东方玄幻风格
- 违禁词替换（杀→陨落/寂灭，死→魂飞魄散/灰飞烟灭，血→猩红/殷红/染血）
- 每章结尾"作者说"互动引导

经两轮优化（初版→补足字数+增加对比+剧情合理性修正），最终定稿10章。

## 3. 手动流水线的真实瓶颈

| 环节 | 实际耗时 | 核心瓶颈 | 自动化可行性 |
|------|----------|----------|-------------|
| 创意生成 | 5轮迭代 | 需要人类判断力（选A还是B、风格偏好、人物关系取舍） | 低——创意方向必须人类决策 |
| 大纲规划 | 1次大输出 | 上下文窗口——240章逐章分解+伏笔追踪，单文件近3000行 | 中——结构化展开可自动化，但伏笔创意需人审 |
| 章节生成 | 10章/轮 | 质量一致性（字数达标、风格锁定、伏笔回收、违禁词替换） | 高——但有质量衰减风险 |
| 质量检查 | 隐含在生成中 | 没有独立检查步骤，依赖事后发现 | 高——可作为独立Agent |

**关键认识**：创意方向必须人类决策；结构化展开（大纲）和草稿生成（章节）可自动化；质量检查应该是独立环节而非隐含步骤。

## 5. 关键原则（本次讨论确立）

1. **人类决策不可替代**：创意方向、风格偏好、人物关系取舍——这些属于创作判断，Agent 提供选项而非代替选择
2. **自动化不等于取代**：工作流的目标是把重复性劳动（大纲展开、章节草稿、伏笔检查）自动化，释放人类精力到创意思考上
3. **校验优于生成**：在长篇小说场景中，检查（是否忘了伏笔、人物状态是否矛盾）比生成更难也更重要
4. **上下文是核心约束**：750章的信息量远超任何单次对话窗口——记忆系统不是可选功能，是核心架构
5. **先跑通再固化**：手动跑完一遍完整流水线，暴露真实瓶颈，再设计自动化的形状——比纸上谈兵有效

## 7. 第二轮讨论：架构设计 — 创作全流程梳理

> 日期：2026-05-03
> 主题：先理清流程再谈分工 — 把整个小说创作的全生命周期画出来

### 7.1 方法论

在定义 Agent 之前，先回答一个问题：**整个小说创作过程到底有哪些环节？** 只有把流程看清楚，才能判断：
- 哪些环节适合由一个 Agent 独立负责
- 哪些环节需要 Agent 与人类深度协作
- 哪些能力是贯穿全流程的基础设施

### 7.2 全流程总图（6阶段 + 3贯穿线）

```
阶段〇                        阶段一                          阶段二
 前创意                        创意定型                        大纲构建
 (想法)                       (8维设定)                      (章节级分解)
─────────────────────────────────────────────────────────────────
用户说：            →  题材·子类型·基调         →  卷级结构
"我想写废材         →  世界观·战力体系           →  阶段→子阶段→情节单元
逆袭玄幻"           →  主角·配角·反派            →  逐章任务分解
                    →  核心冲突·爽点设计          →  伏笔规划与埋设点位
                    →  节奏模型·篇幅规划          →  节奏模式分布
                    →  开篇钩子                   →  境界成长曲线对齐章节

  人类决策密度：最高              高                       中
  Agent角色：脑暴伙伴           结构化展开                系统性分解


阶段三                        阶段四                          阶段五
 章节生成                      质量校验                        持续创作
 (逐章/批量)                   (独立检查)                     (迭代循环)
─────────────────────────────────────────────────────────────────
→ 加载人物状态快照    →  字数是否达标            →  回到阶段三写下一批
→ 加载前情摘要        →  风格是否一致            →  每10-20章抽查
→ 加载本章大纲        →  伏笔是否呼应            →  每卷结束做全卷审计
→ 加载活跃伏笔        →  人物状态是否矛盾        →  大纲偏离时触发预警
→ 生成正文            →  违禁词是否替换          →  必要时回到阶段一调整大纲
→ 保存+更新摘要       →  节奏是否合理
→ 更新人物状态        →  输出：合格/不合格+问题列表
                      →  不合格→退回阶段三重写

  人类决策密度：低             低(自动化)                   中(阶段性审阅)


三条贯穿线（横跨阶段〇到五，不绑定任何单一阶段）
═══════════════════════════════════════════════════════════════════

贯穿线A：记忆管理
  人物状态快照(境界/伤势/物品/关系) → 逐章更新 → 供所有阶段读取
  章节摘要(每章200字) → 上下文压缩 → 阶段三加载前情时使用
  世界观一致性(区域天花板/境界体系/势力分布) → 阶段三校验时比对

贯穿线B：伏笔生命周期
  埋设(阶段一/二规划) → 发展(阶段三写入) → 回收(阶段三写入)
  → 追踪(阶段四检查) → 逾期未收告警 → 阶段二调整大纲插入回收点

贯穿线C：风格锁定
  人称/视角/语气 → 违禁词表 → 爽点密度标准 → 描写要素清单
  → 阶段三输入 → 阶段四校验
```

### 7.3 各阶段详细展开

#### 阶段〇：前创意（起点）

```
输入：用户的一句话想法
示例："我想写一个废材逆袭流的玄幻小说"

过程：
  1. Agent 提问题，帮用户把模糊想法具体化
     - "你想写单女主、多女主还是无女主？"
     - "节奏偏向纯爽文，还是爽中有虐？"
     - "有没有参考作品？"
  2. 用户回答，Agent 继续收敛
  3. 产出：一个足够具体的创作方向描述（半页纸）

输出：方向描述 → 进入阶段一

人类角色：唯一的决策者
Agent角色：提问者、选项提供者
```

#### 阶段一：创意定型（8维设定）

```
输入：阶段〇的方向描述

过程（参考 fiction-crafter 的8维框架）：
  1. 题材定位（主类型/子类型/风格基调/目标篇幅）
  2. 世界观设定（力量体系/势力分布/区域规则）
  3. 主角人设（身份/性格/金手指/成长弧线）
  4. 主要配角（女主/导师/队友/反派，各自的动机和弧线）
  5. 核心冲突（主线矛盾/支线矛盾/感情线矛盾）
  6. 爽点体系（小爽点/中爽点/大爽点的分布和触发条件）
  7. 节奏架构（因果链模型/节奏模式交替规则）
  8. 开篇钩子（第一章如何抓住读者）

特点：
  - 不是一次性输出，是迭代收敛
  - 每轮：Agent提方案 → 人类提修改意见 → Agent修订
  - 8个维度之间有依赖关系（比如世界观中的区域天花板决定了篇幅分布）
  - 实战中经历了5轮迭代才定稿

输出：完整创意提示词（类似 提示词.md）
     包含：全部8维设定 + 待确认事项清单

人类角色：每轮给出修改方向（"双女主""节奏不能单一""东荒需要扩展"）
Agent角色：根据反馈修订方案，标记矛盾点（"如果东荒只有180章，5境跨度会过密"）
```

#### 阶段二：大纲构建

```
输入：阶段一的完整创意提示词

过程：
  1. 卷级拆分（按区域/境界线/核心冲突分期）
  2. 每卷的阶段拆分（3-4个阶段，每个阶段有明确的境界区间和核心悬念）
  3. 每阶段的子阶段拆分（3-4个子阶段，每个有剧情主题）
  4. 每子阶段的情节单元拆分（3-5个情节单元，每个2-6章）
  5. 每个情节单元的节奏标注和伏笔埋设点位
  6. 伏笔追踪表的初始化（每条伏笔：名称/埋设章/预期回收章/类型）
  7. 境界成长曲线与章节的对应校验（"第100章突破灵海境→前99章是否有足够积累"）

特点：
  - 前三卷可以做逐章级分解，后几卷可以只做到阶段级
  - 树状结构：卷→阶段→子阶段→情节单元→章
  - 每一步展开都要检验因果链是否连贯
  - 伏笔需要规划"埋设→发展提示→回收"的完整生命周期

输出：全局大纲文件 + 伏笔追踪表 + 境界-章节对照表
     第一卷：逐章任务分解
     后续卷：阶段级分解（待接近时再展开到逐章）

人类角色：审阅大纲结构，确认节奏和篇幅分布合理
Agent角色：树状展开，因果链校验，标注矛盾
```

#### 阶段三：章节生成

```
输入：
  - 本章大纲（来自阶段二）
  - 人物状态快照（来自贯穿线A）
  - 前情摘要（来自贯穿线A，前3-5章的内容摘要）
  - 活跃伏笔清单（来自贯穿线B，本章需要呼应或发展的伏笔）
  - 风格设定（来自贯穿线C）

过程：
  1. 加载输入 → 组装 prompt
  2. 生成正文（2200-2500字）
  3. 生成后：更新人物状态（境界变化/新增伤势/获得物品/关系变化）
  4. 生成本章摘要（200字以内）→ 存入记忆
  5. 标记伏笔状态（埋设/发展/回收）

特点：
  - 可以单章生成，也可以批量（5-10章一批）
  - 批量的风险是质量漂移——所以需要阶段四独立检查
  - 每次生成时只加载"需要的"上下文，不是全部750章记忆
  - 实战中的教训：批量生成后才发现字数不达标，说明批量需要有中间检查点

输出：章节正文.md + 更新后的状态快照 + 本章摘要

人类角色：不参与单章，批量后抽查
Agent角色：纯执行
```

#### 阶段四：质量校验

```
输入：阶段三生成的章节 + 质量检查清单

检查维度：
  1. 字数检查：是否在2200-2500字（硬性指标）
  2. 风格检查：是否为第一人称、是否违反违禁词表
  3. 伏笔检查：本章是否有应该呼应但遗漏的活跃伏笔
  4. 状态检查：人物境界/伤势/物品是否与记忆中的状态一致
     （例：第5章断了肋骨→第6章不能忽然完好无损）
  5. 节奏检查：本章的实际节奏是否符合大纲标注的节奏模式
  6. 逻辑检查：战斗结果是否符合战力体系
     （跨境而战需有金手指或计谋支撑，不能凭空越级）

输出：
  合格 → 标记为已校验，进入存储
  不合格 → 退回阶段三，附带具体问题列表：
    "字数不足：当前1800字，需补充400-700字"
    "人物状态矛盾：第5章林云断了两根肋骨，本章开篇未提及伤势恢复"
    "伏笔遗漏：第4章埋设的'太上长老感知'本章无任何呼应"

人类角色：不参与（除非反复不合格触发人工介入）
Agent角色：独立校验，不与Writer共享生成上下文——这是核心设计原则
```

#### 阶段五：持续创作与迭代

```
三层循环结构：

小循环（每章/每批）：
  阶段三（生成）→ 阶段四（校验）→ 合格则下一批

中循环（每10-20章）：
  人类抽查 → 发现风格漂移或节奏问题 → 调整阶段三的参数

大循环（每卷结束）：
  全卷审计：
    - 伏笔回收率（已埋X条，已回收Y条，超期未收Z条）
    - 节奏分布是否符合阶段二的规划
    - 境界成长曲线是否与大纲一致
    - 人物弧光是否完整
  → 如果发现大纲需要调整 → 回到阶段二修订大纲
  → 如果发现创意设定有问题 → 回到阶段一修订设定
```

### 7.4 反馈回路

```
阶段〇 ←──────────────────────────── 阶段五发现的创意级问题
阶段一 ←──────────────────── 阶段五大审计发现设定需要修订
阶段二 ←──────────── 阶段三/四发现大纲不合理（比如章节不够用）
阶段三 ←── 阶段四退回不合格章节
```

反馈的远近取决于问题层级：
- **章节级问题**（字数不足、单一伏笔遗漏）→ 阶段四→阶段三，最短回路
- **大纲级问题**（篇幅不足、节奏分布偏离）→ 阶段五→阶段二，中等回路
- **设定级问题**（战力体系有漏洞、世界观矛盾）→ 阶段五→阶段一，最长回路

#### Agent 1：创意顾问（Creative Advisor）

| 维度 | 内容 |
|------|------|
| 对应流程阶段 | 阶段〇（前创意）+ 阶段一（创意定型） |
| 输入 | 用户的初始想法（一句话或一段话） |
| 输出 | 完整8维创意提示词 |
| 核心能力 | 提问收敛模糊想法、8维框架展开、迭代修订、标记维度间矛盾 |
| 对应现有技能 | fiction-crafter（8维生成）+ 需要补充交互协议 |
| 人机交互密度 | **最高**——每轮迭代需要人类反馈，创意方向必须人类决策 |
| 不可自主决定的事项 | 女主策略、风格基调、篇幅规模、人物关系走向 |

#### Agent 2：大纲规划师（Outline Planner）

| 维度 | 内容 |
|------|------|
| 对应流程阶段 | 阶段二（大纲构建） |
| 输入 | 已确认的创意提示词 |
| 输出 | 卷→阶段→子阶段→情节单元→章的树状分解 + 伏笔追踪表初始化 + 境界-章节对照 |
| 核心能力 | 树状结构展开、因果链校验、伏笔生命周期规划、节奏分布计算 |
| 对应现有技能 | 无直接对应（my-novel-writer 有大纲管理命令但无自动展开能力） |
| 人机交互密度 | **中**——产出后需人类审阅确认结构合理性 |

#### Agent 3：章节生成器（Chapter Writer）

| 维度 | 内容 |
|------|------|
| 对应流程阶段 | 阶段三（章节生成） |
| 输入 | 本章大纲 + 人物状态快照 + 前情摘要 + 活跃伏笔清单 + 风格设定 |
| 输出 | 2200-2500字正文 + 更新后的人物状态 + 本章摘要 |
| 核心能力 | Prompt 组装、风格锁定、违禁词替换、生成后状态更新 |
| 对应现有技能 | my-novel-writer 的 `--generate` 命令（部分可用） |
| 人机交互密度 | **低**——批量生成后仅抽查 |
| 关键约束 | 每次只加载所需上下文切片，不加载全部记忆 |

#### Agent 4：质量检查官（Quality Inspector）

| 维度 | 内容 |
|------|------|
| 对应流程阶段 | 阶段四（质量校验） |
| 输入 | 生成的章节 + 质量检查清单 + 记忆管理器中的当前状态 |
| 输出 | 合格/不合格 + 具体问题列表（字数/风格/伏笔/状态/节奏/逻辑） |
| 核心能力 | 6维独立检查，不与 Writer 共享生成上下文 |
| 对应现有技能 | 无直接对应——这是实战中确认缺失的关键角色 |
| 人机交互密度 | **低**——自动化运行，仅反复不合格时触发人工介入 |
| 设计原则 | **必须独立于 Writer**——自己写的东西自己查不出问题 |

#### Agent 5：伏笔追踪器（Foreshadowing Tracker）

| 维度 | 内容 |
|------|------|
| 对应流程阶段 | 贯穿线B + 阶段五（大审计） |
| 输入 | 所有已写章节 + 伏笔追踪表 |
| 输出 | 伏笔状态报告（已埋未收/已回收/超期未收）+ 告警 |
| 核心能力 | 周期性扫描、语义匹配（前文设定是否在后续出现）、逾期告警 |
| 对应现有技能 | onkos 的语义检索能力可部分复用（需要封装伏笔匹配逻辑） |
| 人机交互密度 | **最低**——仅在发现逾期未收伏笔时通知人类 |
| 运行模式 | 不与生成流程耦合——每写完一个阶段后批量扫描 |

### 8.2 底层基础设施：记忆管理器（Memory Manager）

记忆管理器不是 Agent，而是共享数据层。每个 Agent 在运行时只加载它需要的切片。

```
novel_memory/
```

### 8.3 三个待决策问题

#### 决策 1：Agent 数量 —— 5个 vs 3个

| 方案 | 内容 | 优点 | 缺点 |
|------|------|------|------|
| A（5 Agent） | Advisor / Planner / Writer / Inspector / Tracker 全拆分 | 职责清晰，Inspector 独立于 Writer 提供真正的质量保障 | 编排复杂度高，Agent 间通信量大 |
| B（3 Agent） | Advisor / Planner+Writer 合并 / Inspector+Tracker 合并 | 编排简单，通信开销小 | Writer 自检不可靠（实战已验证），Planner 和 Writer 共享上下文可能超出窗口 |

**实战教训**：手动流程中，大纲展开和章节生成是同一个"我"在执行，导致没有独立校验环节，字数不达标的问题直到第一轮全部写完才被发现。

#### 决策 2：复用现有技能 vs 新建 Agent

| 技能 | 可否直接复用 | 问题 |
|------|-------------|------|
| fiction-crafter | ✅ 可复用为 Advisor | 目前只有创意生成，缺少"选项呈现"和"迭代反馈"的交互协议 |
| my-novel-writer | ⚠️ 部分复用 | `--generate` 可用，但 memory 结构需要扩展（目前只有 novel_context.json，不够细粒度） |
| onkos | ⚠️ 部分复用 | 语义检索可用于 Tracker，但需要封装伏笔匹配逻辑 |
| prompt-optimizer | ❌ 不适合 | 6阶段分析是为通用提示词优化设计的，不是为小说工作流 |

**建议方向**：复用 fiction-crafter 和 my-novel-writer 的核心能力，但需要写一个薄的 orchestration 层串联它们。Inspector 和 Tracker 需要从零构建——但逻辑相对独立，可以各自封装。

#### 决策 3：人类审批节点

| 节点 | 审批方式 | 理由 |
|------|----------|------|
| 创意提示词产出后 | **必须审批** | 创意方向一旦确定，后续所有产出都基于此——这里不能错 |
| 每卷大纲产出后 | **必须审批** | 结构性问题（节奏、篇幅、境界分布）需要人类判断 |
| 每10-20章 | **抽查** | 不需要逐章审批，但需要阶段性验证风格和质量没有漂移 |
| Inspector 标记"不合格"时 | **自动退回 Writer** | 不需要人类介入——这是纯质量问题 |

## 10. 第三轮讨论：基于 inkos 参考的优化架构 v2

> 日期：2026-05-03
> 主题：结合对 @actalk/inkos（10-Agent CLI）和 onkos（Claude Code skill）的深入学习，重新优化整个小说写作工作流
> 方法论：不照搬，取 inkos 验证过的设计模式，融合我们实战中发现的真实瓶颈

### 10.1 从 inkos 验证的核心原则

研读 inkos v1.3.7 源码后，以下设计选择被其工程实践强力验证：

| 原则 | inkos 中的体现 | 我们的采纳 |
|------|---------------|-----------|
| **审/改/润分离** | Auditor → Reviser → Polisher 三个独立 Agent | ✅ 采纳——这是"自己查不出自己问题"的延伸 |
| **结构化章节指令** | ChapterMemo（7段 YAML frontmatter） | ✅ 采纳——替代散装"本章大纲" |
| **受控上下文** | Planner→Composer→Writer，Writer 不直接读真相文件 | ✅ 采纳——解决上下文膨胀和信息越界 |
| **结构化状态变更** | Writer 输出 RuntimeStateDelta，不是自由格式 | ✅ 采纳——状态更新可追溯可回滚 |
| **卷级压缩** | Consolidator 压缩章节摘要为卷级叙事段落 | ✅ 采纳——长篇小说上下文管理的核心机制 |
| **类型预设** | 15 个 Genre Profile（审计维度/疲劳词/节奏规则） | ✅ 采纳——先用玄幻类型落地 |
| **快照回滚** | 每章边界创建完整状态快照 | ✅ 采纳——支持安全的重写和回退 |
| **问题分类路由** | LOCAL（措辞修复）vs STRUCTURAL（重写） | ✅ 采纳——避免小修引出大bug |
| **Hook 依赖链** | dependsOn / coreHook / halfLife / promoted | ✅ 采纳——替代简单的三状态模型 |

#### 为什么是 6 Agent（不是 5 也不是 10）？

| 拆分点 | inkos 做法 | 我们的选择 | 理由 |
|--------|-----------|-----------|------|
| 创意 vs 大纲 | 拆（Architect + Planner） | **合**（Advisor 覆盖阶段〇+一，Planner 覆盖阶段二） | 创意定型和大纲构建之间有明确的"人类审批"边界，天然分离 |
| 规划 vs 上下文选择 | 拆（Planner + Composer） | **合**（Planner 同时输出 ChapterMemo + ContextPackage） | Composer 的上下文选择规则可内嵌到 Planner 的 prompt 中，不增加 LLM 调用 |
| 写 vs 提取事实 | 拆（Writer 写 + Observer 提取） | **合**（Writer 一步输出正文 + StateDelta） | inkos 拆是因为用不同 temperature——我们可以在一次 Claude 调用中完成 |
| 审计 vs 修订 vs 润色 | 拆（Auditor + Reviser + Polisher） | **拆**（三个独立 Agent） | **这是最关键的分拆**——审/改/润三者职责冲突，合并会导致质量退化 |

### 10.3 ChapterMemo：章节级结构化写作指令

这是 inkos 最值得借鉴的单个设计。我们当前的"本章大纲"是自由文本，Writer 容易偏离核心任务。ChapterMemo 用 7 段结构化 YAML 锁定本章的完整约束。

#### 格式定义

```yaml
---
chapter: 11
title: 兽潮来袭
goal: 林云在兽潮中首次公开展露实力，引起坊市注意——但不过度暴露混沌珠
rhythm: 爆发
wordCount: 2200-2500
pov: 第一人称（林云）
mood: 紧张中带着克制——林云不想暴露但不得不出手
```

#### Planner 生成 ChapterMemo 的 Prompt 结构

```
输入：
  - Volume outline node（本章在树状结构中的位置）
  - 前 3 章 ChapterMemo（了解近期节奏和伏笔状态）
  - 活跃伏笔清单（从 Tracker 获取）
  - 人物当前状态快照（从 Memory 获取）
  - Genre Profile（节奏规则、疲劳词表、爽点类型）

输出要求：
  - 严格遵循 7 段 YAML 格式
  - 每个 [ ] 条目必须是可验证的（"展示战力"不可验证→应改为"展示开元巅峰的灵力化形雏形"）
  - Held-Back 段是必填的——每章必须明确"不写什么"
  - Don'ts 必须包含疲劳词限制
  - 解析失败重试最多 3 次

关键约束：
  - 活跃伏笔上限 = 12 条（超过则必须在本章回收或延期部分伏笔）
  - 节奏模式必须与大纲标注一致
```

### 10.4 Governed Context：受控上下文协议

这是 inkos v2 模式的核心——Writer 不能直接读取全部真相文件，只能看到 Planner 为其预选的上下文切片。这解决了两个问题：
1. **上下文膨胀**：750 章的信息量远超单次对话窗口
2. **信息越界**：Writer 可能无意中写入角色不该知道的信息

#### ContextPackage 格式

```json
{
  "chapter": 11,
  "assembledBy": "Planner",
  "selectedContexts": [
    {
      "source": "characters/林云.json",
      "fields": ["currentRealm", "injuries", "knownTechniques", "activeRelationships"],
      "reason": "本章林云战斗，需当前战力和伤势"
    },
    {
      "source": "characters/田老板.json",
      "fields": ["currentRealm", "personality", "relationshipWith.云尘"],
      "reason": "本章救田老板，需确认角色关系"
    },
    {
      "source": "chapter_summaries/ch009.json",
      "fields": ["summary", "endingState"],
      "reason": "坊市设定来源——本章在同一坊市"
    },
    {
      "source": "chapter_summaries/ch010.json",
      "fields": ["summary", "newHooks", "characterStateChanges"],
      "reason": "上章结尾：混沌珠第一层开启，开元巅峰，需要衔接"
    },
    {
      "source": "pending_hooks.json",
      "filter": "payoffChapter <= 15",
      "reason": "近期需兑现或推进的伏笔"
    },
    {
      "source": "world.json",
      "fields": ["realmSystem.开元境", "东荒.坊市", "兽潮设定"],
      "reason": "本章涉及开元巅峰战斗和兽潮"
    },
    {
      "source": "style_profile.json",
      "fields": ["forbiddenWords", "dialogueStyle", "descriptionDensity"],
      "reason": "风格锁定"
    }
  ],
  "excludedContexts": [
    {"source": "characters/沈清雪.json", "reason": "本章不出场"},
    {"source": "characters/姬瑶.json", "reason": "尚未出场"},
    {"source": "chapter_summaries/ch001-005.json", "reason": "已超出上下文窗口"},
    {"source": "world.json>中州", "reason": "东荒阶段不需要中州设定"}
  ]

```

#### 上下文选择规则（硬编码，不依赖 LLM）

```
规则 1：POV 过滤
  - 本章 POV 角色 = 林云 → 只加载林云视角可获得的信息
  - 排除：其他角色在林云不在场时发生的事件

规则 2：时间窗口
  - 必须加载：前 3 章摘要
  - 条件加载：前 4-10 章中埋设有本章需回收伏笔的章节
  - 不加载：前 10 章以外的详细摘要（已压缩为卷摘要）

规则 3：伏笔临近度
  - payoffChapter 在 [当前章, 当前章+5] 范围内 → 必须加载
  - payoffChapter 在 [当前章+6, 当前章+20] → 可选加载（仅标题）
  - payoffChapter > 当前章+20 → 不加载（到时再加载）

规则 4：人物相关度
  - 本章出场人物 → 加载完整状态
  - 被提及但不出场 → 仅加载名称和关系概要
  - 不出场且不被提及 → 不加载

规则 5：世界设定
  - 本章涉及的地域/境界/势力 → 加载
  - 远期才涉及 → 不加载
```

### 10.5 质量保障三阶段：审计 → 修订 → 润色

这是 v2 架构最重要的结构改进。inkos 的 Auditor → Reviser → Polisher 三阶段分离被其工程实践验证为正确选择——审、改、润三者职责冲突，合并在一个 Agent 中会导致质量退化。

#### 阶段 A：质量审计官（Continuity Auditor）— 40维综合审计

##### 10.5.1 设计原则

v2 初版 22 维审计直接取自 inkos 玄幻配置的子集，过于片面。现综合 **四个来源** 的全部审计能力，重新设计：

| 来源 | 审计维度数 | 执行方式 | 独特贡献 |
|------|-----------|----------|----------|
| **inkos** | 37 维（玄幻启用~22维） | LLM | 12条读者雷点、备忘偏离、Hook-debt升级规则、信息越界、读者期待管理 |
| **onkos** | 6 维 + 4 连续性子检查 | Python 脚本 | 感官分布、对话标签多样性、句首重复、段落长度方差、位置矛盾检测 |
| **fiction-crafter** | 9 维全审 + 情节漏洞清单 | LLM + 模板 | 人物成长停滞检测、桥段重复相似度、有效信息比、历史趋势追踪 |
| **my-novel-writer** | 5 类生成时约束 | 脚本 + Prompt | 字数硬校验、违禁词替换、章末钩子检测、QUALITY_CHECKLIST |

**核心设计原则**：

1. **能脚本化的绝不调用 LLM** — onkos 的确定性检查零成本，应作为第一道过滤网
2. **LLM 审结构不审文笔** — 借鉴 inkos 的 Reviewer 边界：文笔归 Polisher，结构归 Auditor
3. **稀疏 Memo 不判 incomplete** — 借鉴 inkos：喘息章/过渡章的 Memo 可以只有 goal + 骨架
4. **Hook-debt 升级门控** — 借鉴 inkos Phase 7：只有 promoted=true 的伏笔才能触发 critical
5. **维度可配置** — 借鉴 inkos 的 Genre Profile：不同题材启用不同维度子集

##### 10.5.3 三层执行架构

```
输入：章节正文 + ChapterMemo + ContextPackage + 状态快照 + 伏笔台账
  │
  ├─ Layer 1: 脚本化审计（13维，零LLM成本，~1秒完成）
  │    维度：24, 27, 30, 31, 32, 33, 34, 35, 39
  │    输出：确定性 PASS/FAIL + 具体定位（段落号、行号、次数）
  │
```

##### 10.5.6 评分体系

借鉴 inkos 的评分校准 + fiction-crafter 的四级评级 + onkos 的 health_score 算术平均：

```
Layer 1 脚本层：每维 0-100（确定性评分）
  → health_score = Σ(script_scores) / 脚本维数

Layer 2 混合层：每维 0-100（LLM确认后赋分）
  → hybrid_score = Σ(hybrid_scores) / 混合维数

Layer 3 LLM层：每维 0-100（LLM审计赋分）
  → llm_score = Σ(llm_scores) / LLM维数

综合评分：
  overallScore = 0.15 × health_score + 0.20 × hybrid_score + 0.65 × llm_score

评级映射：
  overallScore ≥ 90 → EXCELLENT（可直接发布）
  overallScore ≥ 80 → GOOD（有小瑕疵，可发布）
  overallScore ≥ 70 → NEEDS_REVISION（需修订）
  overallScore < 70  → NEEDS_REWRITE（需重写）

passed = (不存在 CRITICAL 级问题) AND (overallScore ≥ 70)
```

### 10.6 Hook 全生命周期模型 v2

基于 inkos 的 Hook 模型升级。我们原来的"埋设→发展→回收"三状态太粗糙。

```
Hook 状态机：
  seed(种子) → planted(已埋设) → hint(提示) → progressing(推进中) 
  → partial_resolve(部分回收) → resolved(已回收)
  ↓                    ↓
  deferred(延期)       abandoned(废弃)

Hook 元数据：
  - id: 唯一标识
  - description: 伏笔内容
  - type: character(人物) / plot(剧情) / setting(设定) / item(物品)
  - priority: critical(核心3-7条) / normal / minor
  - coreHook: true/false（全书中只有3-7条核心伏笔）
  - dependsOn: [hook_id, ...]（此伏笔揭晓依赖的其他伏笔）
  - plantedChapter: 埋设章
  - expectedPayoffChapter: 预期回收章
  - payoffTiming: immediate(同章) / near_term(1-5章) / mid_arc(6-20章) / slow_burn(21-50章) / endgame(卷末+)
  - halfLifeChapters: 半衰期章数（超过此数未推进则衰减urgency）
  - advancedCount: 推进次数
  - promoted: true/false（推进次数≥2自动升格）
  - lastAdvancedChapter: 最近推进章
  - urgency: 0-5（基于半衰期和超期计算）
  - status: 当前状态
  - resolution: 回收描述（回收后填写）
```

**Urgency 计算公式**（借鉴 onkos 的数学衰减模型）：

```
base = (priority == critical ? 3 : priority == normal ? 2 : 1) 
     * (1 + 0.15 * advancedCount)

chaptersSinceLastAdvance = currentChapter - lastAdvancedChapter

if chaptersSinceLastAdvance > halfLifeChapters:
    urgency = base * (1 + 0.3 * (chaptersSinceLastAdvance - halfLifeChapters) / 10)
else:
    urgency = base * max(0.25, 1.0 - chaptersSinceLastAdvance * 0.04)

Clamp to [0, 5.0]
```

**核心伏笔（coreHook）规则**：
- 全书 3-7 条核心伏笔
- 核心伏笔不得跨卷不推进（每卷至少推进 1 次）
- 核心伏笔逾期 > 50 章未推进 → CRITICAL 告警
- 普通伏笔逾期 > 30 章未推进 → HIGH 告警
- 次要伏笔（minor）允许自然衰减到 0 后自动废弃

**Planner 的伏笔预算感知**：
- 活跃伏笔上限 = 12 条（超过则 Planner 必须在 Memo 中安排回收或标记延期）
- 每章最多新埋 2 条伏笔
- 每章应至少推进 1 条已有伏笔

```
  1. 按阶段（4阶段）分组摘要
  2. 每阶段用 LLM 压缩为一段 500 字叙事段落
  3. 保留关键数据：境界变化节点、核心伏笔推进节点、主要人物状态变化
  4. 合并为 volume_summary（~2000 字覆盖 240 章）

输出：
  - volume_summaries.md 新增该卷段落
  - 详细摘要归档到 summaries_archive/
  - chapter_summaries.md 仅保留当前卷的行

效果：
  写第241章时，上下文中的摘要从 240 行（~48000字）压缩到 1 个卷段落（~2000字）
  + 当前卷（中州）的详细行逐步累积
```

### 10.8 Genre Profile：玄幻类型预设

借鉴 inkos 的 15 个 genre 文件设计。这是将我们在实战中积累的规则集中化的配置文件——不再分散在多个 Agent 的 prompt 中。

```markdown
# Genre Profile: 东方玄幻（废材逆袭流）

## 基础设定
- 叙事视角：第一人称（林云 POV）
- 目标篇幅：~750 章（东荒240 + 中州220 + 天道170 + 超脱120）
- 每章字数：2200-2500 字
- 语言：简体中文，东方玄幻风格

## 审计维度（40 维全启用，按章节类型动态选择子集）
默认启用：全部40维（玄幻类）。快速审模式（过渡章/缓冲章）：25维。详见第10.5.8节配置表。

## 疲劳词表（出现 ≥3 次/章触发警告）
冷笑, 蝼蚁, 倒吸凉气, 瞳孔骤缩, 不可置信, 轰然炸裂, 
满场死寂, 难以置信, 仿佛, 不禁, 宛如, 竟然, 恐怖如斯,
此子, 天骄, 妖孽, 变态, 区区, 找死, 不知死活

## 违禁词表（出现直接 HIGH 级告警）
杀 → 陨落/寂灭/斩落
死 → 魂飞魄散/灰飞烟灭/陨落
血 → 猩红/殷红/染血/赤红
尸 → 遗骸/残骸/躯壳
鬼 → 阴灵/冥魂

## 节奏规则
- 三章内必有明确反馈：打脸、收益兑现、信息反转、地位变化——四者至少其一
- 每 10 章内至少有 1 个大爽点（四要素齐全：对手之强+差距之悬+临阵突破+真实代价）
- 缓冲章（纯日常/修炼）不得超过连续 2 章
- 每卷必须有 1 次境界突破

## 爽点类型（6 类）
1. 打脸：被低估→反差展露→旁观者震惊
2. 升级突破：积累到位→破境→新能力展示
3. 收益兑现：前文投入→本章收获（物品/身份/信息）
4. 智斗碾压：以智取胜→对手事后才明白
5. 身份揭示：隐藏身份或实力被揭开的瞬间
6. 底牌亮出：隐藏能力首次展示

## 战力体系规则
- 跨境战斗必须满足至少 2 项：金手指加成 / 计谋 / 环境利用 / 对手状态不佳 / 付出代价
- 同一卷内跨境跨度 ≤ 2 境（炼体不能跨境打灵海）
- 境界突破必须有铺垫：修炼积累 ≥ 3 章 / 战斗领悟 / 外部机缘
- 战力数值必须有账本（particle_ledger）：灵石消耗、丹药使用、功法熟练度

## 禁止事项（全流程）
- 禁止主角突然心软放过该杀的敌人
- 禁止主角无缘由的顿悟（必须有前文铺垫）
- 禁止反派排队送人头（每个反派应有独立动机）
- 禁止无代价的能力觉醒
- 禁止"恰好路过"式巧合（每次巧合必须有因果链支撑）
- 禁止描写跳过战斗（搜索/掠夺场景避免清单式罗列）
- 禁止用"暴涨""海量"等抽象词替代具体修炼描写

## 金手指约束（混沌珠专用）
- 能力天花板：混沌珠共7层，每层有明确的上限
- 使用代价：每次激活需要消耗神识/灵力/寿命/其他
- 触发条件：每层开启需要特定条件（如第一层需要"濒死+修复"）
- 成长路径：层数对应境界（第1层→开元，第2层→凝魂……第7层→帝境）
```

### 10.9 完整流水线：一章从规划到定稿的全旅程

以下是一章（以第11章为例）经历的完整流程：

```
═══════════════════════════════════════════════════════════════════════
STEP 0: PRE-PLAN（人类触发或自动触发）
═══════════════════════════════════════════════════════════════════════

  触发条件（任一满足）：
    - 上一章已通过审计+润色+落定
    - 人类发出"继续写下一章"指令
    - Daemon 模式：scheduler 触发

═══════════════════════════════════════════════════════════════════════
STEP 1: PLAN（大纲规划师 — LLM 调用）
═══════════════════════════════════════════════════════════════════════

  输入：
    ✓ volume_map.md（找到第11章的节点定义）
    ✓ 前3章 ChapterMemo（第8-10章）
    ✓ 活跃伏笔清单（从 hooks.json，筛选 payoffChapter <= 15）
    ✓ 人物状态快照（从 current_state.json）
    ✓ Genre Profile（节奏规则、疲劳词表、爽点类型）
    ✓ 伏笔预算（当前活跃数 / 12 上限）

  处理：
    1. 检查伏笔预算 → 活跃伏笔 = 9，可以新埋 2 条 ✓
    2. 解析大纲节点 → 第11章属于"东荒·阶段一·身份建立·兽潮危机"
    3. 确定节奏模式 → 爆发（大纲标注）
    4. 填充 7 段 ChapterMemo
    5. 组装 ContextPackage（按5条硬规则选择上下文）
    6. 验证：Memo 中的任务是否都有可验证标准？

  输出：
    ✓ ChapterMemo（7段 YAML）
    ✓ ContextPackage（选定上下文 + 排除列表 + 选择理由）

═══════════════════════════════════════════════════════════════════════
STEP 2: WRITE（章节生成器 — LLM 调用）
═══════════════════════════════════════════════════════════════════════

  输入：
    ✓ ChapterMemo（完整7段）
    ✓ ContextPackage（仅选中的上下文，非全部真相文件）
    ✓ Style Profile（统计指纹 + 风格指南）
    ✓ Genre Profile（违禁词表、禁止事项）

  PRE_WRITE_CHECK（生成前自检——嵌入 prompt）：
    □ 本章核心目标是什么？→ ChapterMemo.goal
    □ 本章绝对不能写什么？→ ChapterMemo.donts
    □ 本章必须扣留什么？→ ChapterMemo.heldBack
    □ 本章必须兑现什么？→ ChapterMemo.mustKeep

  生成：
    - Temperature: 0.7（创作模式）
    - 按 ChapterMemo 的 Task List 逐项推进
    - 每写完一个场景，自检是否触犯 Don'ts

  生成后提取（同一 LLM 调用，temperature 降到 0.3）：
    ✓ 事实提取 → 供 StateDelta 使用
    ✓ 状态变更识别 → "林云新增轻伤""田老板认知升级"
    ✓ 伏笔活动标记 → "推进了hook_003""新埋设hook_017"

  输出：
    ✓ 章节正文（2200-2500字）
    ✓ RuntimeStateDelta（结构化状态变更）
    ✓ 本章摘要（200字，7列格式）

═══════════════════════════════════════════════════════════════════════
STEP 3: AUDIT（质量审计官 — LLM 调用）
═══════════════════════════════════════════════════════════════════════

  输入：
    ✓ 章节正文
    ✓ ChapterMemo（检查备忘偏离）
    ✓ ContextPackage（检查信息越界）
    ✓ 人物状态快照（检查状态矛盾）
    ✓ 活跃伏笔清单（检查伏笔遗漏）
    ✓ Genre Profile（审计维度选择）

  审计（40维，分三层执行——脚本层13维 + 混合层8维 + LLM层19维）：
    逐维检查 → 每维 PASS/FAIL + score 0-100

  问题分类（硬编码路由）：
    词汇疲劳、违禁词、段落形态 → LOCAL
    OOC、战力崩坏、备忘偏离、主线偏离、伏笔遗漏 → STRUCTURAL

  输出：
    ✓ AuditReport（整体评分 + 分维评分 + 问题列表 + 备忘偏离分析）

═══════════════════════════════════════════════════════════════════════
STEP 4: DECISION GATE（硬编码，无 LLM 调用）
═══════════════════════════════════════════════════════════════════════

  ┌─ verdict = PASS → 跳转到 STEP 6（POLISH）
  │
  ├─ verdict = PASS_WITH_WARNINGS → 检查 issues 类型：
  │    ├─ 全部 LOCAL → STEP 5a（LOCAL PATCH）
  │    └─ 含有 STRUCTURAL → STEP 5b（REWRITE）
  │
  └─ verdict = FAIL → STEP 5b（REWRITE）

═══════════════════════════════════════════════════════════════════════
STEP 5a: LOCAL PATCH（修订官 — LLM 调用）
═══════════════════════════════════════════════════════════════════════

  约束：
    - 只修改 AuditReport.issues 中标记的位置
    - 不新增/删除人物或情节
    - 字数变化不超过 ±50 字

  输出：
    ✓ 修补后的章节
    → 返回 STEP 3 重新审计（仅检查被修改的维度）

═══════════════════════════════════════════════════════════════════════
STEP 5b: REWRITE（修订官 — LLM 调用）
═══════════════════════════════════════════════════════════════════════

  使用：
    - 相同的 ChapterMemo
    - 相同的 ContextPackage
    - Temperature: 0.5 + (retryRound * 0.1)

  策略：
    - 保留原章中未被标记为问题的段落
    - 仅重写被标记为 STRUCTURAL 的段落及其上下文

  输出：
    ✓ 重写后的章节
    → 返回 STEP 3 重新审计

  迭代上限：
    - 最多 3 轮（第1轮改写 + 第2轮改写 + 第3轮改写）
    - 保留每轮快照，取最高分版本
    - 3轮后仍不合格 → 标记为"需人工介入"

═══════════════════════════════════════════════════════════════════════
STEP 6: POLISH（润色官 — LLM 调用）
═══════════════════════════════════════════════════════════════════════

  前置条件：AuditReport.verdict = "PASS"

  输入：
    ✓ 已审计通过的章节
    ✓ Style Guide + Style Profile
    ✓ Genre Profile（违禁词表再次确认）

  操作（仅文笔层面）：
    - 句子节奏优化（长短句交替）
    - 对话自然度提升
    - AI腔移除（套话、书面语腔调）
    - 五感描写强化（缺少感官锚点的段落）
    - 标点/段落间距统一

  禁止：
    ✗ 改动任何情节、角色台词含义、人物/事件/设定

  输出：
    ✓ 润色后章节
    ✓ [polisher-note] 列表（发现结构问题时记录但不修改）

═══════════════════════════════════════════════════════════════════════
STEP 7: SETTLE（落定 — 脚本操作，无 LLM 调用）
═══════════════════════════════════════════════════════════════════════

  操作：
    1. 应用 RuntimeStateDelta → 更新 current_state.md / current_state.json
    2. 更新伏笔 → hooks.json / pending_hooks.md
    3. 写入章节摘要 → chapter_summaries.md / chapter_summaries.json
    4. 创建快照 → snapshots/ch011/
    5. 更新角色档案 → roles/林云.md 等
    6. 验证状态一致性（StateValidator）：
       - 所有被引用的实体都在事实表中
       - 伏笔的 dependsOn 指向有效伏笔
       - 人物状态与最近摘要一致
    7. 标记章节为"已落定"

  输出：
    ✓ 所有状态文件更新完成
    ✓ 快照已创建
    ✓ 一致性验证通过
    ✓ 就绪，等待下一章触发

═══════════════════════════════════════════════════════════════════════
每 10-20 章：人类抽查
═══════════════════════════════════════════════════════════════════════

  人类审阅最近 10-20 章：
    - 风格是否漂移？
    - 节奏是否符合预期？
    - 是否有需要调整的创作方向？
    → 如有问题 → 调整 Genre Profile 或 ChapterMemo 模板参数

═══════════════════════════════════════════════════════════════════════
每卷结束：全卷审计 + 压缩
═══════════════════════════════════════════════════════════════════════

  STEP V1: 全卷审计（必须人类审批）
    - 伏笔回收率（已埋X / 已回收Y / 超期Z）
    - 节奏分布 vs 大纲规划
    - 境界成长曲线 vs 章节对应
    - 人物弧光完整性
    → 如偏离 → 回到 Planner 调整后续大纲

  STEP V2: Consolidator 卷级压缩
    - 按阶段（4阶段）分组摘要
    - LLM 压缩为叙事段落
    - 归档详细摘要
    - 释放上下文空间

  STEP V3: 卷边界快照
    - 完整状态快照（卷级）
    - 标记卷结束
```

### 10.11 三个决策的答案

之前第 9 节悬而未决的三个决策，通过学习 inkos 已自然消解：

#### 决策 1：Agent 数量 —— 6 Agent + 2 基础设施（+1 可选）

```
✅ 采纳 6 Agent（介于原方案 A 的 5 和 B 的 3 之间，取 inkos 验证过的拆分点）

拆分理由（每个拆分都有 inkos 的工程实践支撑）：
  Advisor / Planner 合？→ 不合——阶段〇+一（创意）和阶段二（大纲）之间有必须人类审批的边界
  Planner / Writer  合？→ 不合——Planner 制定计划，Writer 执行计划，职责天然分离
  Writer / Auditor  分？→ 必须分——这是最重要的分离，"自己写的东西自己查不出问题"
  Auditor / Reviser 分？→ 必须分——审计是诊断，修订是治疗，诊断不清就下药会治错
  Reviser / Polisher分？→ 必须分——修订改结构，润色改文笔，职责冲突
  Tracker 独立？     → 是——伏笔需要跨所有阶段运作，不绑定任何 Agent
  Consolidator 独立？→ 是——卷级压缩是独立的长篇小说基础设施
```

#### 决策 2：复用策略 —— "复用核心能力 + 新建编排层 + 升级 onkos 脚本"

```
✅ 三层策略：

第一层（直接复用，不改代码）：
  - fiction-crafter：8维创意生成 + 迭代修订 → Advisor
  - onkos/entity_extractor.py：实体提取
  - onkos/semantic_model.py：语义搜索
  - onkos/style_learner.py：风格分析

第二层（复用核心逻辑，封装新接口）：
  - my-novel-writer/--generate：章节生成核心 → Writer 的 LLM 调用层
  - onkos/continuity_checker.py：确定性检查 → Auditor 的快速预检层
  - onkos/hook_tracker.py：基础伏笔管理 → Tracker 的数据层
  - onkos/memory_engine.py：6级摘要 + FTS5 → Memory 的存储层

第三层（全新构建）：
  - Orchestrator：流水线编排、Step间数据传递、Gate决策
  - ChapterMemo 生成逻辑（Planner 的核心能力）
  - Governed Context 协议（ContextPackage 组装和校验）
  - Auditor 的 40维三层审计系统（脚本层 + 混合层 + LLM层）
  - Reviser 的 LOCAL/STRUCTURAL 路由和迭代控制
  - Polisher 的文笔润色 prompt
  - Consolidator 的卷级压缩逻辑
```
