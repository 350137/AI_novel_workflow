## 三、Agent 架构（5 Agent + 3 基础设施）

### 3.1 为什么是 5 Agent

inkos 有 8+ Agent，onkos 有 8 个角色。我们精简为 5 Agent，每次合并都有明确理由：

| 合并决策 | inkos/onkos中分离 | 我们合并为 | 理由 |
|---------|------------------|----------|------|
| Architect + Planner | inkos: Architect(基础设计) + Planner(逐章Memo) | **hjw-novel-planner** | 串行执行，人类审批在中间，温度差异小(0.5-0.7) |
| Composer → Planner | inkos: Composer(组装ContextPackage) | **hjw-novel-planner** | ContextPackage是大纲的自然延伸，不需要独立Agent |
| Writer 保持独立 | — | **hjw-novel-writer** | 自己写自己审=医生给自己开刀，绝对不可合并 |
| Auditor 保持独立 | — | **hjw-novel-auditor** | 诊断和治疗分离是inkos验证的核心原则 |
| Reviser 保持独立 | — | **hjw-novel-reviser** | 与Auditor职责不同(诊断≠治疗)，合并会导致"知道问题但改不干净" |
| Polisher 保持独立 | — | **hjw-novel-polisher** | 结构修改与文笔打磨冲突，一起做会互相污染 |

**必须分离的边界**（不可合并）：
- **Writer ≠ Auditor**：自己写自己审 → 盲区
- **Auditor ≠ Reviser**：诊断 ≠ 治疗 → 互相妥协
- **Reviser ≠ Polisher**：结构 ≠ 文笔 → 互相污染

## 四、各步骤详解（整合最佳实践）

### 第一步：头脑风暴 → 完善创意

**要解决的问题**：把"我想写个废材逆袭玄幻"这种模糊想法，收敛成可执行的完整创作方案。

```
输入：一句话想法
      │
      ▼
┌─────────────────────────────────────────┐
│ 收敛性提问                                │
│  来源：fiction-crafter(8维) + inkos(5段)  │
│         + tomato-novelist(6问爆款)        │
│                                          │
│  · 题材定位：主类型/子类型/风格基调/篇幅     │
│  · 世界观：力量体系/势力分布/区域规则        │
│  · 主角人设：身份/性格/金手指+代价/成长弧线  │
│  · 配角框架：各有独立动机和弧线             │
│  · 核心冲突+爽点+节奏+开篇钩子             │
│  · 情绪标签：爽文/虐恋/甜宠/悬疑           │
└────────────────┬────────────────────────┘
                 │ 人类每轮反馈
                 ▼
┌─────────────────────────────────────────┐
│ 8维创意提示词 + Genre Profile 初始化       │
│  ①题材·基调  ②世界观·战力                │
│  ③主角人设  ④配角框架                    │
│  ⑤核心冲突  ⑥爽点设计                    │
│  ⑦节奏规划  ⑧开篇钩子                    │
│                                          │
│ 同步产出：Genre Profile                   │
│  · 疲劳词表 · 违禁词替换 · 节奏规则        │
│  · 爽点类型 · 战力体系规则 · 禁止事项      │
└────────────────┬────────────────────────┘
                 │
       ⚠️ 人类审批（必过门控）
                 │
                 ▼
输出：完整创意提示词 + Genre Profile
```

**有据可依的最强贡献**：

| 来源 | 贡献到这个步骤 |
|------|-------------|
| fiction-crafter | 8维框架 + 主动标记维度间矛盾（"东荒180章，5境太密"） |
| inkos | 5段散文式 foundation（story_frame / volume_map / roles / book_rules / pending_hooks） |
| tomato-novelist | 6问锁定爆款基因 + 情绪标签 + 核心反差定位（扮猪吃虎/马甲掉落/追妻火葬场…） |
| novel-writer-cn | 10类型全覆盖 + 4种结局变体预设 |
| openclaw-novel-write | AI自动分析类型/风格 + 用户确认门控 |

**Agent：hjw-novel-advisor（创意顾问）**
- 提问者、方案生成者——**绝不替人类做创意决定**
- 主动标记维度间矛盾
- 不可自主决定：女主策略、风格基调、篇幅规模、人物关系走向

### 第二步：世界观+大纲 → 结构化展开

**要解决的问题**：把创意方案展开为可逐章执行的树状大纲，确保世界观自洽、节奏可控。

```
输入：创意提示词 + Genre Profile
      │
      ▼
┌─────────────────────────────────────────┐
│ 卷级规划（来源：inkos OKR 递归大纲）       │
│  总目标 → 卷目标 + 3 Key Results         │
│  按区域/境界线/冲突分期划分卷              │
│  每卷3-4阶段，各有境界区间和核心悬念       │
└────────────────┬────────────────────────┘
                 │
                 ▼
┌─────────────────────────────────────────┐
│ 逐章分解                                  │
│  阶段 → 子阶段 → 情节单元 → 章            │
│  每章：ChapterMemo（7段YAML）             │
│  节奏模式标注：爆发/蓄压/后效/过渡          │
└────────────────┬────────────────────────┘
                 │
                 ▼
┌─────────────────────────────────────────┐
│ 同步产出                                  │
│  · 伏笔表（种→埋→暗示→推进→回收）          │
│  · 境界-章节对照校验                      │
│  · 节奏分布（爆发/蓄压/过渡比例）          │
│  · ContextPackage（每章受控上下文切片）     │
│  · 爽点节奏公式（小爽1-2章/中爽3-5章/     │
│    大爽8-12章/卷末15-20章）               │
└────────────────┬────────────────────────┘
                 │
       ⚠️ 人类审批（每卷大纲产出后）
                 │
                 ▼
输出：卷级大纲 + 逐章 ChapterMemo + 伏笔台账 + ContextPackage
```

**ChapterMemo 7段结构**（来源：inkos Planner）：

```yaml
---
chapter: N
title: ""
goal: ""            # ≤50字
rhythm: "爆发|蓄压|后效|过渡"
wordCount: "2200-2500"
pov: "第一人称（林云）"
mood: ""
---

## 1. 本章任务（Task List）
## 2. 必须兑现（Payoffs / Must-Keep）
## 3. 本章扣留（Held-Back）
## 4. 日常/过渡功能（Daily/Transition Function）
## 5. 三问测试（3-Question Test）
## 6. 章末状态变更（End-of-Chapter State Changes）
## 7. 本章禁止事项（Don'ts）
```

**有据可依的最强贡献**：

| 来源 | 贡献到这个步骤 |
|------|-------------|
| inkos | OKR递归大纲 + ChapterMemo格式 + 一角色一文件 + Governed Context |
| fiction-crafter | 卷级章节规划表（任务/爽点/情节/情绪曲线/状态）+ 套路重复预检 |
| tomato-novelist | 情绪曲线模板 + 10种钩子技巧预先分配 |
| story-writer | 三幕结构(25%-50%-25%) + 5种情节模式 |
| onkos | 6级层次摘要 + 事实重要性分级 |

**Agent：hjw-novel-planner（大纲规划师）**
- 树状展开：卷→阶段→子阶段→情节单元→逐章Memo
- 伏笔预算：活跃上限12条，每章最多新埋2条
- 境界成长曲线对齐章节
- 前三卷逐章分解，后面卷阶段级分解
- Planner 14条内化原则（来源：inkos）

### 第三步：依据大纲 → 逐章写作

**要解决的问题**：把 ChapterMemo 变成 2200-2500 字的正文，同时追踪所有状态变化。

```
输入：ChapterMemo + ContextPackage + Style Profile + Genre Profile
      │
      ▼
┌─────────────────────────────────────────┐
│ PRE_WRITE_CHECK（来源：my-novel-writer）  │
│  □ 核心目标？ □ 绝不能写什么？             │
│  □ 必须扣留什么？ □ 必须兑现什么？         │
└────────────────┬────────────────────────┘
                 │
                 ▼
┌─────────────────────────────────────────┐
│ 生成正文（temperature 0.7）               │
│                                          │
│ 遵守约束：                                │
│  · 6条句法红线（来源：novelist）           │
│    ① 最多1个比喻/页                      │
│    ② 肯定句式为主，不用"没有不"           │
│    ③ 短句+句号，动作不插破折号            │
│    ④ 禁止全知形而上学总结                 │
│    ⑤ 中文语感，不能西式从句堆叠            │
│    ⑥ 不用"他知道/感觉到/意识到"           │
│                                          │
│  · 违禁词自动替换（my-novel-writer）      │
│  · 章末钩子（10种轮换，tomato-novelist）  │
│  · 五感交替描写（onkos）                  │
│  · 对话占比30-50%（onkos）                │
│  · 2200-2500字硬约束                      │
│  · 推拉镜头法则：微观感官→宏观场景（novelist）│
└────────────────┬────────────────────────┘
                 │
                 ▼
┌─────────────────────────────────────────┐
│ 提取结构化输出（来源：inkos + onkos）      │
│  · StateDelta（事实/伏笔/人物状态变更）    │
│  · 本章摘要（200字，7列格式）              │
│  · 隐藏情节追踪卡（来源：novelist）        │
└────────────────┬────────────────────────┘
                 │
                 ▼
输出：章节正文 + StateDelta + 本章摘要
```

**有据可依的最强贡献**：

| 来源 | 贡献到这个步骤 |
|------|-------------|
| novelist | **6条句法红线**（最硬核的风格约束）+ 心理锚点剥离 + 推拉镜头 + 黑名单词汇 |
| inkos | Governed Context（Writer不见全貌，只读ContextPackage）+ StateDelta格式 |
| my-novel-writer | 写前自检 + 字数强制扩充(内心独白/环境/动作/旁观反应) + 违禁词替换表(杀→陨落) |
| tomato-novelist | 10种开篇钩子 + 10种章末钩子轮换 + 情绪标签场景 |
| onkos | 五感交替描写 + 对话占比30-50% + show-don't-tell |

**Agent：hjw-novel-writer（章节生成器）**
- 只读 ContextPackage，不加载全部真相文件（防止信息越界）
- Temperature 0.7
- 输出正文 + StateDelta + 摘要

### 5.5 评分与路由

```
评分公式：
  overallScore = 0.15 × scriptHealth + 0.20 × hybridScore + 0.65 × llmScore
  passed = (no CRITICAL issues) AND (overallScore ≥ 70)

评级标准：
  95-100  可发布
  85-94   轻微瑕疵
  75-84   有明显问题
  65-74   多项问题
  <65     结构性崩溃

问题分类与路由：
  CRITICAL → STRUCTURAL → 重写（保留未被标记段落）
  WARNING  → LOCAL      → 原地修补（措辞替换/疲劳词/违禁词/段落调整）
  INFO     → LOCAL      → 记录，不阻塞通过
```

## 六、基础设施（三条贯穿线）

### 6.1 Memory（记忆管理）
来源：onkos（6级摘要+SQLite）+ inkos（markdown真相文件+事实生命周期）

```
6级摘要体系：
  全书(200字) → 阶段(200字) → 弧(150字) → 卷(500字) → 章(200字,7列表格) → 场景(100字)

事实分级：
  permanent（永久）—— 核心设定，始终加载
  arc_scoped（弧级）—— 当前卷内有效
  chapter_scoped（章级）—— 最近3章有效

存储结构：
  story/     —— Markdown 真相文件（人类可读）
  state/     —— JSON 结构化状态（程序读写）
  snapshots/ —— 每章快照（支持回滚）
  summaries_archive/ —— 已压缩卷的详细摘要归档
```

### 6.2 Tracker（伏笔追踪）
来源：inkos（完整状态机+promoted+dependsOn+halfLife）+ onkos（urgency公式）

```
状态机：
  seed → planted → hint → progressing → partial_resolve → resolved
                    ↓                    ↓
                   deferred            abandoned

核心字段：id, description, type, priority, coreHook, dependsOn,
  plantedChapter, expectedPayoffChapter, payoffTiming(5级),
  halfLifeChapters, advancedCount, promoted(advancedCount≥2自动升格),
  lastAdvancedChapter, urgency(0-5,半衰期衰减公式), status, resolution

预算控制：活跃上限12条，每章新埋≤2条
债务升级：仅promoted=true的coreHook逾期可触发CRITICAL
```

### 6.3 Consolidator（卷级压缩）
来源：inkos Consolidator

```
每卷结束后触发：
  输入：该卷所有章摘要 + 伏笔台账
  处理：LLM压缩为~2000字叙事段落
  输出：存储到 summaries_archive/，释放上下文窗口
  副作用：重算伏笔promoted状态
```

## 七、流水线（一章的完整旅程）

```
STEP 1: PLAN（Planner）
  读大纲节点 → 读前3章Memo → 读活跃伏笔 → 读人物状态
  → 生成本章 ChapterMemo（7段YAML）
  → 组装 ContextPackage（5条硬规则）

STEP 2: WRITE（Writer）
  PRE_WRITE_CHECK：
    □ 核心目标？ □ 绝不能写什么？ □ 必须扣留什么？ □ 必须兑现什么？
  → 生成正文（temperature 0.7，2200-2500字）
  → 提取 StateDelta → 生成摘要

STEP 3: AUDIT（Auditor）
  Layer 1（脚本，13维）→ Layer 2（混合，8维）→ Layer 3（LLM，19维）
  → 输出 AuditReport（overallScore + verdict + issues）

STEP 4: DECISION GATE（硬编码路由）
  PASS（≥90，无CRITICAL）→ POLISH
  PASS_WITH_WARNINGS / FAIL
    ├─ 全部LOCAL → REVISE（原地修补）
    └─ 含有STRUCTURAL → REVISE（重写）
    → 回 STEP 3 重新审计（最多3轮）
    → 3轮仍FAIL → ⚠️ 人工介入

STEP 5: POLISH（Polisher — 仅当审计通过）
  ✅ 句子节奏/对话自然度/AI腔移除/五感强化
  ❌ 禁改情节/台词含义/人物/事件/设定

STEP 6: SETTLE（落定）
  应用StateDelta → 更新状态 → 更新伏笔 → 写摘要 → 创建快照 → 验证一致性
```

## 八、反馈回路

```
章节级问题（字数不足/单一伏笔遗漏）
  → 第四步→第三步，最短回路

大纲级问题（篇幅不足/节奏分布偏离）
  → 卷末审计→第二步，中等回路

设定级问题（战力体系漏洞/世界观矛盾）
  → 卷末审计→第一步，最长回路

持续迭代循环：
  小循环（逐章）：Plan → Write → Audit → Revise/Polish → Settle → 下一章
  中循环（每10-20章）：人类抽查 → 发现偏差 → 调整参数
  大循环（每卷结束）：⚠️ 全卷审计 → 偏离则回第二步调整大纲
```

## 九、关键设计原则

| 原则 | 说明 | 来源 |
|------|------|------|
| **审/改/润分离** | Auditor诊断→Reviser治疗→Polisher打磨，职责不混 | inkos |
| **Writer不见全貌** | Writer只读ContextPackage，不加载全部真相文件 | inkos Governed Context |
| **脚本优先LLM** | 能用正则/jieba/统计的检查不调LLM（40维中13维脚本化） | onkos |
| **Hook升级门控** | 只有promoted=true的核心伏笔能触发CRITICAL | inkos Phase 7 |
| **稀疏Memo豁免** | 喘息章/过渡章不因Memo内容少而扣分 | inkos |
| **快照可回滚** | 每章落定后创建完整状态快照 | inkos + onkos |
| **人类在关键节点** | 创意产出、每卷大纲、每卷末全审——三个必须审批点 | 实战验证 |
| **句法红线不可破** | 6条硬句法规则，违者必改（来源：novelist） | novelist |
