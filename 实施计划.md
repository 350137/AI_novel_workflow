### 任务 1.5：Hook v2 数据模型

**产出**：`protocols/hook_model.schema.json` + `scripts/hook_manager.py`

**Hook 状态机**（已在设计文档第10.6节定义）：
```
seed → planted → hint → progressing → partial_resolve → resolved
                    ↓                    ↓
                   deferred            abandoned
```

**核心字段**：id, description, type, priority, coreHook, dependsOn, plantedChapter, expectedPayoffChapter, payoffTiming, halfLifeChapters, advancedCount, promoted, lastAdvancedChapter, urgency, status, resolution

**hook_manager.py 功能**：
- `plant_hook(...)` — 埋设新伏笔
- `advance_hook(hook_id, chapter)` — 推进伏笔（advancedCount+1，检测是否触发 promoted）
- `resolve_hook(hook_id, chapter, resolution)` — 回收伏笔
- `defer_hook(hook_id, reason)` — 延期伏笔
- `compute_urgency(hook_id, current_chapter)` — 计算 urgency（基于半衰期公式）
- `check_hook_budget()` — 检查活跃伏笔是否超过12条上限
- `get_active_hooks(filter)` — 按条件筛选活跃伏笔

**验收**：Hook 状态机所有转换路径正确；urgency 计算正确；promoted 自动升格正确

### 任务 2.5：Reviser（修订官）

**产出**：`agents/reviser.prompt.md`

**输入**：原始章节 + AuditReport

**路由逻辑**（硬编码，不依赖LLM）：
- 全部 LOCAL → 原地修补（措辞替换/疲劳词/违禁词/段落调整）

**迭代控制**：
- 最多3轮，每轮 temperature +0.1
- 保留每轮快照，选最高分版本
- 3轮后仍不合格 → 标记为需人工介入

**验收**：能根据 AuditReport 正确路由 LOCAL vs STRUCTURAL，修补不引入新问题

## Phase 3：流水线编排

> 目标：用 Orchestrator 把所有 Agent 和脚本串成可运行的流水线
> 依赖：Phase 2
> 产出：orchestrator.py + 核心数据传递逻辑

### 任务 3.1：Orchestrator 主脚本

**产出**：`scripts/orchestrator.py`

**功能**：
- `write_chapter(chapter_number)` — 运行完整的一章流水线（Plan → Write → Audit → Revise/Polish → Settle）
- `write_batch(start_chapter, count)` — 批量生成多章
- `rewrite_chapter(chapter_number)` — 从快照回滚并重写指定章
- `audit_only(chapter_number)` — 仅审计指定章（用于抽查）
- `get_status()` — 返回当前进度

**流水线状态机**：
```
PLAN → WRITE → AUDIT → DECISION_GATE
                            ├─ PASS → POLISH → SETTLE → DONE
                            ├─ LOCAL_ONLY → REVISE_LOCAL → AUDIT
                            ├─ STRUCTURAL → REVISE_STRUCTURAL → AUDIT
                            └─ MAX_RETRIES → NEEDS_HUMAN
```

**Gate 决策逻辑**（硬编码）：
- `verdict = PASS` → 跳转到 POLISH
- `verdict = PASS_WITH_WARNINGS` + 全部 LOCAL → REVISE_LOCAL
- `verdict = PASS_WITH_WARNINGS` + 含有 STRUCTURAL → REVISE_STRUCTURAL
- `verdict = FAIL` → REVISE_STRUCTURAL
- 第3次 AUDIT 仍 FAIL → NEEDS_HUMAN

**验收**：能跑通单章完整流水线；状态机和 Gate 决策正确

```
  Input:  volume_map.md + 前3章 ChapterMemo + hooks.json + current_state.json + genre_profile.md
  Output: ChapterMemo.md + ContextPackage.json

Step 2 WRITE:
  Input:  ChapterMemo.md + ContextPackage.json + style_profile.json + genre_profile.md
  Output: chapter_N.md + StateDelta.json + chapter_summary.json

Step 3 AUDIT:
  Input:  chapter_N.md + ChapterMemo.md + ContextPackage.json + hooks.json + current_state.json
  Output: AuditReport.json

Step 4 DECISION:
  Input:  AuditReport.json
  Route:  → REVISE / POLISH / NEEDS_HUMAN

Step 5 REVISE:
  Input:  chapter_N.md + AuditReport.json + ChapterMemo.md + ContextPackage.json
  Output: chapter_N_revised.md
  Loop:   → AUDIT (重新审计)

Step 6 POLISH:
  Input:  chapter_N.md + style_guide.md + genre_profile.md
  Output: chapter_N_final.md + polisher_notes.json

Step 7 SETTLE:
  Input:  chapter_N_final.md + StateDelta.json + chapter_summary.json
  Ops:    更新所有状态文件 + 创建快照 + 运行 StateValidator
  Output: 状态文件已更新 + 快照已创建
```

**验收**：每个 Step 的输入/输出格式被所有相关 Agent 和脚本遵守

**步骤**：

## 待决策事项

开始实施前，需要确认以下问题：

### 决策 A：实施起点

当前最自然的起点是 **Phase 0.2（数据迁移）**——把我们已写好的第1-10章、提示词、大纲全部导入记忆系统。这既是搭建基础设施，也是为后续验证做准备。

### 决策 B：Agent 调用方式

所有 Agent prompt 模板定义好后，调用方式有两种：
- **方案 1**：每个 Agent 作为独立的 Claude Code 子会话（`Agent` tool），prompt 文件作为 system context
- **方案 2**：Agent 作为当前会话中的角色切换（加载不同的 prompt 到当前上下文）

建议方案 1（子会话隔离），理由：Auditor 必须独立于 Writer 的生成上下文。

### 决策 C：脚本语言

onkos 现有脚本是 Python。我们新增的脚本用 Python 以保持一致性，还是用 Node.js 以匹配 inkos 生态？

建议 Python（与 onkos 统一，jieba/onnxruntime 已在环境中可用）。
